Authentification & Sécurité

Navigation & accès aux modules

Le CRM est une Single Page Application (index.html). La navigation se fait via la barre latérale gauche (sidebar). Depuis la refonte Pipeline (2026), les anciennes vues "Contacts" et "Contrats" séparées ont été remplacées par un Pipeline Kanban unifié (🗂️ Pipeline) qui regroupe les deux en une seule vue Kanban. Les vues Tâches, Résultats, Messages et Administration restent distinctes.

Espace Prospection Terrain : L'espace /prospection-terrain/index.html est distinct du CRM principal. Accès par code PIN à 4 chiffres (sessionStorage pt_unlocked) — aucune authentification Supabase requise, accessible à tout commercial sur tablette. La topbar affiche le bouton ← CRM pour revenir au CRM. Sidebar organisée en 3 actes : 🚀 Démarrage (ciblage SIRENE) · 🗺️ Lancer l'itinéraire (tournée optimisée) · 🔍 Faire un audit gratuit (missions RGPD / SEO / Cyber).

Connexion & Gestion de session

Connexion email / mot de passe
Authentification sécurisée via Supabase Auth. Session automatiquement expirée après 4 heures (RGPD Art. 42 — déconnexion automatique, alerte à T-15 min).
Double authentification (2FA/TOTP)
QR code scannable (Google Authenticator, Authy). Code 6 chiffres à chaque connexion. Trust 2h après validation. Enrôlement obligatoire si absent.
Mot de passe oublié
Lien de réinitialisation envoyé par email. Renouvellement obligatoire tous les 45 jours avec avertissement anticipé.
Verrouillage automatique
Compte bloqué après 5 tentatives de connexion échouées. Déblocage manuel par l'administrateur.

Audit & Journalisation

  • Enregistrement de chaque tentative de connexion (réussie ou échouée) via l'Edge Function record-login-failure
  • Journal consultable par l'admin : 50 dernières tentatives avec date, email, résultat (✓/✗), indicateur de verrouillage et adresse IP
  • Alertes d'intrusion automatiques : détection des essais multiples et création d'alerte rouge dans le panel admin
  • Résolution manuelle des alertes par l'administrateur (bouton « Résolu ✓ »)

Isolation des données — Row Level Security (RLS)

L'isolation des données entre commerciaux est appliquée simultanément à deux niveaux : dans l'interface et directement dans la base de données PostgreSQL via des politiques RLS, rendant tout contournement par API impossible même avec un token valide.

Fonctions SQL de sécurité (SECURITY DEFINER)

is_admin()
Retourne true si profiles.is_admin = true pour l'utilisateur courant. Utilisée dans toutes les policies d'écriture et de suppression.
is_dci()
Retourne true si profiles.role = 'dci'. Permet au DCI d'accéder en lecture aux fiches de son équipe.
get_team_ids()
Retourne le tableau des id de tous les profils dont dci_parent_id = auth.uid(). Utilisée dans la policy SELECT pour délimiter l'équipe visible du DCI.
Policy SELECT (contacts, contrats, tâches)
is_admin() OU created_by = auth.uid() OU (is_dci() ET created_by IN get_team_ids()). Aucun accès direct en dehors de ces trois cas.
Vérification live : les policies actives ont été auditées directement via pg_policies — aucune policy ouverte (USING true) n'est présente sur les tables contacts, contracts ou tasks. La hiérarchie Niveau 1 / DCI / Admin est intégralement enforced au niveau PostgreSQL.

Clause de Confidentialité

  • Collaborateurs CRM : signature numérique bloquante à la première connexion — si aucune clause signée n'est trouvée dans clauses_confidentialite, le CRM redirige automatiquement vers /clause-confidentialite.html?redirect=/. Retour automatique après signature (3 s). PDF archivé dans Supabase Storage et envoyé par email (template Brevo ID 7). L'action est logguée : clause_confidentialite_signee — Critique.
  • Signataires externes / publics : page dédiée sans authentification, inclut le numéro ORIAS. PDF archivé et envoyé (template Brevo ID 6).
  • Intégralité du document consultable en mode lecture seule via paramètre URL ?view=id.

Contrat de Mandat Agent Commercial

Signature électronique en 4 étapes : lecture du contrat complet → renseignement des informations mandataire → signature canvas → validation OTP par email. SHA-256 du texte du contrat calculé et enregistré. PDF archivé et email envoyé automatiquement avec toute la documentation commerciale (template Brevo ID 5). L'action est logguée : mandat_signe — Critique. Accessible via /mandat.html ou /Force-de-vente/mandat.html (redirection).

Gestion des Contacts

Création & édition d'un contact

Identité complète
Nom, prénom, téléphone, email, LinkedIn. Dénomination entreprise, forme juridique, SIRET.
Adresse avec autocomplétion
Intégration API Gouv (api-adresse.data.gouv.fr). Sélection automatique ville selon code postal.
Activités professionnelles
Cases à cocher : Digitalisation, RGPD, Assurance, Click & Collect, Autre. Utilisées pour le filtrage.
Consentements RGPD explicites
3 consentements distincts : téléphone, email, courrier. Obligatoires pour prospection légale.

Suivi des interactions

  • Types : Téléphone, Email, Visite terrain, LinkedIn, Facebook, Autre
  • Contenu : Type, date, objet de l'échange, compte-rendu détaillé, suite à donner
  • Liens : Associée optionnellement à un contact et/ou un contrat
  • Historique : Chronologique, modifiable, visible dans la fiche contact

Fonctions avancées

  • Transfert de client : Changer le commercial responsable (bouton « Transférer client »)
  • Propriété stricte : Chaque contact appartient à son créateur ; seul le propriétaire ou un super-admin peut le modifier
  • Verrouillage RGPD KO : Coordonnées masquées, édition bloquée si contact en état de non-conformité RGPD
  • Journalisation RGPD : Toute modification sensible est logguée automatiquement
  • Recherche & filtres : Recherche globale (nom, entreprise, email), filtre par activité, filtre par commercial responsable

Résiliation d'abonnement

  • Niveau 1 / DCI : soumet une « Demande de résiliation » — l'admin valide ou rejette

Paiements Stripe
Création d'abonnement
Edge Function create-checkout crée une session Stripe. Lien de paiement généré et transmis au client. Synchronisation automatique du statut contrat après paiement.
Portail client Stripe
Génère un lien temporaire vers le portail Stripe self-service. Le client peut modifier son moyen de paiement, télécharger ses factures, ou résilier lui-même. Lien copié automatiquement dans le presse-papier.
Webhook Stripe
Écoute les événements Stripe : création / mise à jour / suppression d'abonnement, paiement réussi, paiement échoué. Met à jour le statut du contrat dans Supabase en temps réel.
Gestion des impayés
Impayé client = commission DCI de 0 € pour le mois concerné. Aucune rétroactivité sur les commissions déjà versées. S@FE absorbe la perte.

Emails transactionnels Stripe

  • Paiement validé (ID 2) : confirmation envoyée au client avec facture jointe
  • Paiement refusé (ID 1) : alerte envoyée au client avec montant, service, et date de prochain essai

Administration

Panel Admin — 5 onglets

Vue d'ensemble

Totaux cumulés tous utilisateurs depuis le 1ᵉʳ du mois : entrées en contact, CA généré, commissions. Jauges SVG.

Par utilisateur

Grille de tous les utilisateurs avec photos, mini-jauges et codage couleur (rouge/orange/vert selon % objectif). Clic pour voir le détail d'un utilisateur.

Gestion utilisateurs

  • Créer un utilisateur : prénom, email, rôle, mot de passe généré (affiché une seule fois)
  • Modifier les détails du profil : identité, adresse professionnelle, SIRET, région/département/secteur
  • Réinitialiser le mot de passe
  • Bloquer / débloquer un compte
  • Rôles : Niveau 1 (user) · DCI (dci) · Admin (admin) · Super Admin (super_admin)

RGPD — 4 sous-onglets

  • Registre des traitements (Art. 30) : table complète avec finalité, base légale, catégories de données, destinataires, durée de conservation, mesures de sécurité · Export PDF
  • Journal RGPD : log de tous les événements sensibles (création, suppression, export de données personnelles) avec utilisateur et horodatage
  • Demandes de droits : suivi des demandes d'accès, rectification, suppression (Art. 17+) de bout en bout
  • Rapports de conformité : évaluations d'impact et audits automatiques

Sécurité

  • Journal des 50 dernières tentatives de connexion : date, email, résultat, verrouillage, IP
  • Alertes d'intrusion non résolues affichées en bandeau rouge
  • Résolution manuelle des alertes

Réinitialisation des données de test

⚠️ Super Admin uniquement — Irréversible. Supprime tous les contacts, contrats, tâches, interactions, mandats et notifications. Conserve les comptes utilisateurs. Confirmation par frappe de « RÉINITIALISER ».

Génération des documents Force de Vente

Page admin-docs.html accessible depuis le panel admin : génère et uploade en un clic les 8 documents PDF commerciaux dans Supabase Storage (opération ponctuelle, à relancer si mise à jour des documents).

Journal RGPD — Registre des traitements & traçabilité

Architecture du journal

Toutes les actions ayant une incidence sur des données personnelles sont enregistrées dans la table audit_logs (Supabase). Cette table est immuable : un trigger PostgreSQL bloque toute opération DELETE ou UPDATE, conformément à l'article 30 du RGPD (obligation de tenue du registre des activités de traitement). Elle ne peut jamais être purgée, même par un administrateur.

Base légale : RGPD art. 5(2) — Principe de responsabilité (accountability). Art. 30 — Registre des activités de traitement. CNIL délibération 2017-190 — obligation de journaliser les exports de données personnelles.

Première connexion — obligations bloquantes

Deux obligations doivent être satisfaites avant qu'un utilisateur puisse accéder au CRM. Elles sont vérifiées dans loadAll() à chaque démarrage de session :

Clause de confidentialité (checkClauseSigne)
Vérifie l'existence d'une ligne dans clauses_confidentialite pour l'utilisateur courant. Si absente → redirection immédiate vers /clause-confidentialite.html?redirect=/. Après signature, retour automatique au CRM (3 s). S'applique à tous les rôles sauf admin.
Mandat agent commercial (checkMandatSigne)
Disponible mais désactivé par choix produit — la signature du mandat n'est pas forcée au login. Accessible depuis /mandat.html ou /Force-de-vente/mandat.html (redirect).

Pipeline Kanban & RGPD

Le Pipeline Kanban fait des appels directs à Supabase (sans passer par les service layers). Les actions suivantes sont néanmoins tracées dans le journal RGPD :

  • Upload PJ (pj_uploadee — Attention) : nom du fichier, taille, contact_id.
  • Suppression PJ (pj_supprimee — Critique) : nom du fichier, contact_id. La suppression dans Supabase Storage est irréversible.
  • Les déplacements de colonnes, changements de priorité et modifications de checklist ne constituent pas des traitements de données personnelles au sens strict — ils ne sont pas logués.

Accès au journal RGPD

  • Interface dédiée : Onglet Administration → Journal RGPD (réservé aux rôles admin et super_admin).
  • Filtres : période, module, criticité, résultat, recherche libre.
  • Exports : PDF · CSV · Excel — chaque export est lui-même logué (export_journal_pdf / export_journal_csv).
  • La table audit_logs est protégée par RLS : seuls les admins peuvent lire en SELECT. Aucun rôle ne peut DELETE ou UPDATE (trigger immuable).
Référence légale : RGPD art. 30 · CNIL délibération 2017-190 · NIS2 (directive UE 2022/2555) — obligation de traçabilité des accès et actions sur les systèmes traitant des données sensibles. La conservation du journal est illimitée (les logs ne peuvent être supprimés).

Purge automatique des données — RGPD art. 5(1)(e)

La politique de conservation est techniquement enforced par un job pg_cron mensuel dans Supabase. Les données ne sont pas supprimées mais anonymisées (pseudonymisation irréversible), préservant les enregistrements commerciaux et comptables.

Critères d'éligibilité à la purge

CasCondition SQLDuréeAction
Client avec contrats terminés/résiliés Tous les contrats en statut 'Terminé' ou 'Résilié' ET MAX(date_echeance, resilié_at) < NOW() - 5 ans 5 ans Anonymisation contact + interactions
Prospect sans contrat Aucun contrat lié ET contacts.created_at < NOW() - 3 ans 3 ans Anonymisation contact
Contact déjà purgé purge_date IS NOT NULL — Ignoré (idempotent)

Données anonymisées vs. conservées

🗑️ Anonymisées (contact)
  • nom → 'ANONYMISÉ'
  • prenom → NULL
  • email → 'purge-{id}@safe-purge.local'
  • telephone → NULL
  • linkedin → NULL
  • adresse → NULL
  • siret → NULL
  • notes → NULL
  • interactions.contenu → '[contenu anonymisé]'
✅ Conservées
  • Contrats (montants, dates, statuts, PDF)
  • audit_logs (journal immuable)
  • contacts.statut, activites (données statistiques)
  • contacts.purge_date (preuve d'anonymisation)
  • Interactions : type, date, objet

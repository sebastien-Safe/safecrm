📋 Synthèse exécutive

Le présent rapport présente les résultats d'un audit exhaustif du CRM S@FE réalisé en lecture seule sur la base de l'analyse statique du code source (frontend, Edge Functions, schémas SQL, configuration). L'application est une SPA vanilla JS hébergée en statique, s'appuyant sur Supabase (PostgreSQL + Auth + Edge Functions), Stripe (paiements) et Brevo (emails transactionnels).

L'audit couvre 16 domaines et s'appuie sur les référentiels OWASP Top 10, RGPD/CNIL, ANSSI RGS, NIS2, WCAG 2.1 AA et les meilleures pratiques Supabase, Stripe et Deno.

Verdict global

L'application S@FE CRM présente une architecture fonctionnelle et des mécanismes de sécurité pensés (TOTP, RLS, audit logs, signature walls). Cependant, plusieurs vulnérabilités critiques nécessitent une correction immédiate avant toute mise en production à grande échelle : absence de .gitignore exposant potentiellement les credentials, un bug de typo silencieux coupant l'envoi d'emails de facturation, et une CSP trop permissive annulant la protection XSS.

Le code source monolithique (app.js 244 KB, 5 473 lignes) constitue un risque opérationnel majeur — toute modification est difficile à tester et à auditer. L'extraction modulaire en cours est la bonne direction mais doit être accélérée.

Points forts

  • TOTP/MFA intégré et challengé systématiquement à la connexion
  • RLS Supabase activé sur toutes les tables critiques
  • Journal d'activité (audit_logs) avec politiques RLS différenciées admin/user
  • Double signature obligatoire (clause confidentialité + charte SI annuelle)
  • Rate limiting anti-brute-force sur les tentatives de connexion
  • Trigger automatique RGPD pour la durée de conservation des données
  • PWA complète avec Service Worker (offline, manifest, icons)
  • Processus de migration SQL documenté (27 versions)

🔧 Domaine 2 — DevSecOps / CI-CD / Git 45/100

🚨
CRITIQUE — Aucun fichier .gitignore Le projet ne possède pas de .gitignore. Le fichier assets/config.js contenant l'URL Supabase (https://qdjmzietysukediqkebg.supabase.co) et la clé publishable (sb_publishable_8uD3m60n9GfMt2h_GkmB7w_MpdFN5_s) est probablement commité dans l'historique Git. Si le dépôt est public, ou s'il existe un accès unauthorized aux logs Git, ces clés sont exposées. Un attaquant peut interroger la base directement si les RLS présentent des failles. De plus, tout fichier sensible temporaire (logs, exports, scripts) créé à la racine sera automatiquement tracké.
⚠️
Pas de pipeline CI/CD — aucun test automatisé Aucun fichier .github/workflows/, Dockerfile, ou configuration de test (Jest, Playwright…) n'est présent. Les déploiements sont manuels. Toute régression n'est détectée qu'en production.
⚠️
27 migrations SQL appliquées manuellement Les fichiers -schema-v1.sql à -schema-v27.sql sont appliqués à la main sur la base de données Supabase. Aucun outil de migration automatisé (Supabase CLI supabase migration, Flyway…) n'est utilisé. Le risque est une désynchronisation entre l'environnement de développement, de staging et de production.
🟡
README obsolète La documentation de setup mentionne les schémas v2 à v5 alors que la base en est à la version 27. Un développeur entrant suivrait des instructions caduques.
🟡
Scripts utilitaires commités (download-stripe-docs.sh) Des scripts de développement sans rapport avec l'application sont trackés dans le dépôt. Bonne hygiène : les exclure via .gitignore ou les déplacer hors du projet.

🗄️ Domaine 3 — Base de données / Schéma SQL 65/100

⚠️
Schéma initial (v1) avec policies trop permissives Le schéma v1 crée des policies de type auth_full_access_contacts avec USING (auth.role() = 'authenticated') — tout utilisateur authentifié peut lire et écrire toutes les lignes de contacts, contracts, tasks. Les versions suivantes ont raffiné ces politiques, mais si la migration v1 a été appliquée sans toutes les suivantes, ou si une restauration partielle est faite, ces permissions excessives réapparaissent.
⚠️
safe_connectors — policy USING(true) WITH CHECK(true) Schema v26 : CREATE POLICY "auth_safe_connectors" ON safe_connectors TO authenticated USING (true) WITH CHECK (true). Tout utilisateur authentifié peut lire ET écrire n'importe quel enregistrement de cette table, y compris des connecteurs appartenant à d'autres clients. Ce n'est probablement pas l'intention.
🟡
Absence d'index sur les colonnes de filtrage courantes Les migrations n'indiquent pas d'index sur contacts.user_id, audit_logs.created_at, audit_logs.user_id, contracts.statut. Sur un volume croissant, les requêtes RLS (qui filtrent par user_id) peuvent devenir lentes sans index.
✅
Tables d'audit modernes (v27) avec RLS strictes totp_audit et audit_access_log ont des policies restrictives et correctes. La table audit_logs possède deux policies distinctes : admin SELECT all + audit_select_own pour les utilisateurs. Architecture exemplaire.
✅
Trigger RGPD automatique check_rgpd_expiry() — trigger automatique pour la gestion de l'expiration des données personnelles. Bonne pratique de Privacy by Design (Art. 25 RGPD).

⚡ Domaine 4 — Edge Functions (Deno) 55/100

⚠️
Typo critique — répertoire creaye-checkout au lieu de create-checkout La fonction de création de session Stripe est déployée sous le nom creaye-checkout (faute de frappe non corrigée). Tout appel à supabase.functions.invoke('create-checkout', …) retournerait une 404 si le nom n'est pas corrigé aussi côté client. Le risque est une régression silencieuse si quelqu'un renomme le call côté client en corrigeant la faute sans toucher le nom du répertoire déployé.
✅
✅ CORRIGÉ P1-2 — CORS restreint sur toutes les Edge Functions (commit 61fdcda) Les 12 Edge Functions browser-facing ont été mises à jour : "Access-Control-Allow-Origin": "*" → "https://crm.safe-digitalisation.fr". stripe-webhook inchangé (appelé par Stripe, pas le navigateur). Toutes redéployées et actives.
🟡
Imports CDN non épinglés (pas de version exacte de hash) Les Edge Functions importent depuis https://esm.sh/stripe@14, https://esm.sh/@supabase/supabase-js@2 avec des versions majeures uniquement. Un bug introduit dans une version mineure ou patch sera automatiquement récupéré. Les imports devraient utiliser des versions exactes (@14.5.0) ou un lock file Deno.
🟡
Logs d'erreur insuffisants dans les Edge Functions Les blocs catch retournent une 500 générique sans console.error() sur tous les chemins d'erreur. Les incidents de production sont invisibles sans instrumentation supplémentaire.
✅
Vérification JWT Supabase dans send-mandat-email La fonction vérifie le header Authorization: Bearer <token> et valide l'utilisateur via supabase.auth.getUser() avant tout traitement. Bonne pratique.

💳 Domaine 5 — Paiement Stripe 70/100

🚨
CRITIQUE — Webhook Stripe sans vérification de signature si variable absente Dans supabase/function/stripe-webhook/index.ts, la vérification de signature Stripe via stripe.webhooks.constructEventAsync() n'est exécutée que si STRIPE_WEBHOOK_SECRET est défini. Si cette variable d'environnement n'est pas configurée sur le projet Supabase, n'importe qui peut envoyer des événements Stripe falsifiés et déclencher des abonnements, des créations de clients, ou des annulations frauduleuses. La vérification doit être obligatoire, non conditionnelle.
⚠️
Calcul TTC hardcodé — risque légal si TVA change Dans creaye-checkout : Math.round(m * 120) calcule le TTC à 20% (HT × 1.2, exprimé en centimes). Si le taux de TVA applicables évolue (ex. prestations spécifiques, DOM-TOM), la formule devra être mise à jour manuellement. Préférer une variable d'environnement TVA_RATE.
🟡
Numéro ORIAS — incohérence possible assets/app.js affiche ORIAS : N° 26008536 dans les documents générés. À confirmer que ce numéro est identique dans tous les templates (emails Brevo, devis PDF, conditions générales).
✅
5 événements Stripe gérés correctement checkout.session.completed, invoice.paid, invoice.payment_failed, customer.subscription.updated, customer.subscription.deleted sont tous traités avec mise à jour de la base de données. Architecture correcte.

📧 Domaine 6 — Emails / Brevo 65/100

🚨
CRITIQUE — Typo BREBO au lieu de BREVO — tous les emails silencieusement coupés La typo Deno.env.get("BREBO") au lieu de "BREVO" était présente dans 5 Edge Functions : stripe-webhook, send-clause-email, send-password-reset, send-mandat-email, send-crm-email. La variable retournait undefined et la condition if (!BREVO) return; court-circuitait l'envoi silencieusement. Aucun email n'était envoyé depuis la mise en prod : ni facturation, ni reset de mot de passe, ni clauses, ni mandats, ni rapports DPO/SEO/Cyber. ✅ Corrigé — commits ac50d83 + 16932e3 — 5 fonctions redéployées
⚠️
Intégration Brevo via fetch direct sans SDK Les appels à l'API Brevo sont faits via fetch() avec headers manuels. En cas de changement d'API Brevo (authentification, endpoint, format), le code doit être modifié manuellement. Un SDK officiel fournirait une abstraction stable et des messages d'erreur structurés.
🟡
Pas de mécanisme de retry pour les emails transactionnels Si l'appel Brevo échoue (timeout, 5xx), l'email est perdu sans aucune mécanisme de reprise. Les emails de facturation sont des communications légalement importantes.
🟡
IDs de templates Brevo hardcodés Les IDs de templates Brevo (1, 2, 7…) sont écrits en dur dans le code. En cas de recréation des templates dans Brevo, tous les IDs doivent être mis à jour manuellement dans chaque Edge Function.

🔐 Domaine 7 — Authentification / MFA 72/100

✅
TOTP/MFA challengé systématiquement challengeTOTPIfNeeded() est appelé au démarrage de l'application et bloque l'accès si le facteur MFA n'est pas validé. La table totp_audit trace toutes les tentatives.
✅
Anti-brute-force via Edge Function dédiée record-login-failure : 5 tentatives → ban 30 minutes, rate limiting IP à 30 req/heure. Architecture correcte et proportionnée.
✅
Double signature obligatoire avant accès Le mur de signature (_checkAndShowSignatureWall()) vérifie la clause de confidentialité ET la charte SI (valide ≤ 12 mois). Le guide est inaccessible sans les deux. Bonne pratique RGPD et RH.
🟡
TOTP non obligatoire côté serveur (enforcement client-side) Le challenge TOTP est effectué dans le JavaScript client. Un utilisateur qui contournerait le JS (accès direct à l'API Supabase avec un token JWT valide) pourrait accéder aux données sans valider le TOTP. Pour une protection totale, une RLS conditionnelle ou un middleware serveur devrait vérifier l'assurance d'authentification (aal2) du token JWT.
🟡
Réinitialisation de mot de passe par email — vecteur de phishing La fonction send-password-reset envoie un lien de réinitialisation. Si un attaquant peut déclencher un reset pour n'importe quel email, cela constitue un vecteur de harcèlement. Vérifier la présence d'un captcha ou d'un rate limit sur cet endpoint.

👥 Domaine 8 — RBAC / Gestion des rôles 65/100

⚠️
Vérification de rôle admin uniquement côté client isAdmin() retourne !!state.profile?.is_admin — valeur lue depuis la table profiles via Supabase. Si un utilisateur malveillant modifie son propre is_admin à true en base (impossible si la RLS est correcte, mais vérifier), ou si un bug d'assignation de rôle survient, l'UI pourrait afficher des fonctionnalités admin à un utilisateur non-admin. Les Edge Functions critiques (admin-create-user) vérifient le profil côté serveur — c'est bien. Les vérifications d'accès aux vues admin dans l'UI sont purement client-side.
🟡
Pas de rôle intermédiaire — binaire admin / user Le système ne connaît que deux rôles : admin et utilisateur standard. Pour une équipe plus grande, un rôle "manager" (lecture de toutes les données mais pas de création d'utilisateurs) serait utile. Pas critique mais à anticiper.
✅
Création d'utilisateur par Edge Function avec vérification côté serveur admin-create-user vérifie que l'appelant est admin avant de créer l'utilisateur. La logique critique n'est pas uniquement côté client.

🛡️ Domaine 9 — RLS & Sécurité des données 73/100

⚠️
Policy safe_connectors avec WITH CHECK (true) Tout utilisateur authentifié peut insérer ou modifier n'importe quel enregistrement de safe_connectors. Cette table contient probablement des intégrations sensibles (clés API tierces, webhooks). La policy devrait filtrer par user_id = auth.uid() ou par organisation.
🟡
Clé publishable Supabase en clair dans assets/config.js La clé sb_publishable_8uD3m60n9GfMt2h_GkmB7w_MpdFN5_s est volontairement publique (c'est son usage prévu) et ne peut pas accéder aux données sans JWT valide, grâce à RLS. Cependant sans .gitignore, si cette clé est dans l'historique Git d'un dépôt public, les scanners automatiques de secrets (GitHub secret scanning, truffleHog) la signaleront comme exposition.
✅
RLS activé sur toutes les tables critiques Les tables contacts, contracts, tasks, audit_logs, profiles, clauses_confidentialite, chartes_usage_si ont toutes RLS enabled.
✅
Table audit_logs avec double policy granulaire Policy audit_select_own : l'utilisateur ne voit que ses propres logs. Policy admin_select_all : l'admin voit tout. Aucune policy INSERT/UPDATE/DELETE côté client — l'écriture est réservée aux Edge Functions avec service_role.

🔒 Domaine 10 — Cybersécurité (OWASP Top 10) 52/100

🚨
CRITIQUE — CSP avec 'unsafe-inline' pour les scripts Le header Content-Security-Policy de index.html autorise 'unsafe-inline' pour les scripts. Cela annule la protection contre les attaques XSS (Cross-Site Scripting) que la CSP est censée fournir. OWASP A03:2021 — Injection. Si un attaquant parvient à injecter du JavaScript (via un champ de formulaire, un contact malveillant, une URL), ce code s'exécutera sans restriction.
✅
✅ CORRIGÉ P1-1 — SRI ajouté sur tous les CDN (commit 141de8e) Les bibliothèques supabase-js@2.108.2, jsPDF@2.5.1, QRCode@1.5.3, Chart.js@4.4.1 ont été épinglées avec leurs attributs integrity="sha384-…" + crossorigin="anonymous" sur les 12 fichiers HTML. Un hook pre-commit bloque les commits sans SRI. Script scripts/check-sri.sh créé pour vérification manuelle.
⚠️
Service Worker peut cacher du code malveillant injecté sw.js utilise une stratégie cache-first pour les assets JS/CSS. Si un attaquant réussit à faire cacher une version compromise d'un fichier, le Service Worker continuera à servir cette version corrompue même après que le serveur soit nettoyé, jusqu'à la mise à jour du cache (changement de version dans sw.js).
⚠️
Utilisation d'innerHTML — risque XSS résiduel app.js contient 111 occurrences d'escapeHtml et autant d'innerHTML. Si une occurrence d'innerHTML utilise des données non échappées (bug de dev futur), une XSS est possible. La présence de escapeHtml() est une bonne pratique mais sa couverture exhaustive doit être vérifiée par revue de code systématique.
🟡
Pas de headers de sécurité HTTP additionnels Les headers X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy ne sont pas documentés dans la configuration de l'hébergement (GitHub Pages ne permet pas de personnaliser les headers HTTP).
✅
Fonction escapeHtml() systématiquement utilisée 111 appels à escapeHtml() pour toutes les données affichées en HTML. Discipline défensive correcte.

⚡ Domaine 15 — Performance / SEO 48/100

⚠️
app.js 244 KB non minifié — temps de parsing JS élevé Un fichier JS de 244 KB non minifié se traduit par un temps de parsing et compilation significatif sur mobile (Core Web Vitals — Total Blocking Time). Sur un réseau 4G, le chargement seul représente ~800ms. Avec minification + compression Brotli, on atteindrait ~60 KB.
⚠️
3 bibliothèques CDN chargées séquentiellement sans async/defer Supabase JS, jsPDF et QRCode sont chargés en bas de page mais sans async ni defer explicites. Ils bloquent le parsing HTML pendant leur téléchargement.
🟡
Absence de bundler — pas de tree-shaking ni code-splitting Sans Vite, Rollup ou Webpack, tout le code est livré dans un seul bundle non optimisé. Une architecture avec code-splitting permettrait de ne charger que le code nécessaire à la vue active.
✅
PWA complète — Service Worker cache-first La stratégie cache-first pour les assets permet un chargement quasi-instantané lors des visites suivantes. Le manifest est complet avec icônes multi-formats. Bonne expérience utilisateur après la première visite.

🔄 Domaine 16 — Résilience / Continuité d'activité 55/100

⚠️
Pas de stratégie de backup documentée Supabase effectue des backups automatiques (Point-in-Time Recovery sur les plans Pro), mais aucune procédure de restauration n'est documentée. En cas d'incident, combien de temps faut-il pour restaurer ? Qui est responsable ?
⚠️
Single Point of Failure — Supabase cloud L'application dépend entièrement de Supabase cloud. Une panne Supabase (qui survient plusieurs fois par an) rend le CRM totalement inaccessible. Aucun mode dégradé ni fallback n'est implémenté.
🟡
Pas de monitoring / alerting Aucun outil de monitoring applicatif (Sentry, Datadog, ou simple webhook d'alerte) n'est configuré. Les erreurs en production (Edge Functions, échecs de paiement Stripe, erreurs JS) sont invisibles jusqu'à signalement utilisateur.
✅
Service Worker — mode offline partiel Le Service Worker permet une consultation des données mises en cache même sans connexion. L'application reste partiellement fonctionnelle en mode offline pour les opérations de lecture.

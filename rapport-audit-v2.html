03 Trouvailles détaillées
P0 CRITIQUE SEC-01 — Fonction test-brevo déployée en production sans authentification Edge Function

Cause : La fonction test-brevo (slug: test-brevo, version 12, verify_jwt: false) est déployée en production sur Supabase Edge Runtime. Elle est publiquement accessible à /functions/v1/test-brevo sans aucune authentification.

Impact : (1) N'importe qui peut déclencher l'envoi d'emails via les templates Brevo de S@FE (template 5 : Mandat DCI) vers n'importe quelle adresse. (2) En mode ?mode=list, liste tous les templates Brevo actifs — exposition de la structure email de l'entreprise. (3) La clé API Brevo est utilisée de manière non protégée dans la fonction.

Impact métier : Phishing possible au nom de S@FE, spam, atteinte à la réputation, dépassement du quota email Brevo, risque RGPD (envoi d'emails commerciaux sans consentement).

✔️ Fait vérifié — code lu via Supabase MCP
verify_jwt:false public brevo production

Correction : Supprimer immédiatement la fonction via Supabase Dashboard → Edge Functions → test-brevo → Delete. Elle ne doit jamais être redéployée en production.

Complexité : Faible · Temps : 5 min · Rollback : Aucun (suppression uniquement)

P0 CRITIQUE SEC-02 — 36 fonctions SQL admin exécutables par le rôle anon via REST Supabase DB

Cause : Des fonctions PL/pgSQL définies avec SECURITY DEFINER sont accessibles via l'API REST Supabase (/rest/v1/rpc/<fn>) par le rôle anon (non authentifié). Parmi elles : admin_create_user, admin_delete_user, admin_list_users, admin_set_admin, _trigger_login_alert et 31 autres.

Impact : Un attaquant non authentifié peut appeler POST /rest/v1/rpc/admin_list_users pour obtenir la liste complète des utilisateurs CRM, ou tenter d'exécuter admin_create_user / admin_delete_user. Comme ces fonctions sont SECURITY DEFINER, elles s'exécutent avec les privilèges de leur créateur (souvent postgres), contournant le RLS.

Impact métier : Fuite de données utilisateurs, potentiel de création/suppression d'utilisateurs non autorisée, atteinte à la confidentialité (RGPD Art.32).

✔️ Fait vérifié — Supabase advisors (116 entrées analysées)
anon accessible SECURITY DEFINER RPC public

Correction : Pour chaque fonction sensible, révoquer l'accès au rôle anon :
REVOKE EXECUTE ON FUNCTION public.admin_create_user(...) FROM anon, authenticated;
Ou basculer en SECURITY INVOKER si la fonction n'a pas besoin des privilèges élevés.

Complexité : Moyenne · Temps : 2-4h · Migration SQL requise

P1 IMPORTANT SEC-03 — IDOR : agenda-ics expose les tâches de tout utilisateur sans authentification Edge Function

Cause : La fonction agenda-ics (verify_jwt: false) accepte un paramètre ?uid=<user_id> et retourne toutes les tâches non terminées de cet utilisateur au format iCal. Aucune vérification d'identité n'est effectuée.

Impact : Quiconque connaît l'UUID d'un utilisateur CRM peut accéder à l'intégralité de son agenda de tâches. Les UUIDs Supabase sont exposés dans les logs, les interactions, les URL partagées, et dans les données RPC. Insecure Direct Object Reference (IDOR) classique.

Impact métier : Exposition des activités commerciales (rendez-vous, prospects, tâches en cours), violation RGPD Art.5.1.f (intégrité et confidentialité). Les logs confirment 2 UIDs distincts actifs régulièrement.

✔️ Fait vérifié — code + logs d'exécution via Supabase MCP
IDOR verify_jwt:false données personnelles

Correction proposée : Ajouter un token signé HMAC à l'URL (généré par le CRM au moment de la création du lien iCal) et vérifier ce token dans la fonction, OU passer à verify_jwt:true et utiliser le JWT de l'utilisateur pour filtrer ses propres tâches uniquement.

Complexité : Moyenne · Temps : 2h · Impact UX : Le lien iCal existant dans les clients calendrier devra être régénéré

P1 IMPORTANT SEC-04 — Vues SECURITY DEFINER : v_interactions et v_equipe_dci contournent le RLS Supabase DB

Cause : Les vues public.v_interactions et public.v_equipe_dci sont définies avec la propriété SECURITY DEFINER. Elles s'exécutent avec les permissions de leur créateur (superuser ou postgres), ignorant complètement le RLS des tables sous-jacentes.

Impact : Un utilisateur authentifié qui a accès à ces vues peut voir des données de tous les autres utilisateurs, même si les tables interactions et profiles ont des politiques RLS restrictives. Classé ERROR par les advisors Supabase.

Impact métier : Violation potentielle de la séparation des données (un DCI peut voir les interactions d'un autre DCI), atteinte RGPD à la minimisation des données.

✔️ Fait vérifié — advisors sécurité Supabase (niveau ERROR)
SECURITY DEFINER bypass RLS ERROR advisor

Correction : Recréer les vues en SECURITY INVOKER (défaut) :
CREATE OR REPLACE VIEW public.v_interactions WITH (security_invoker = true) AS SELECT ...;
Vérifier que les politiques RLS des tables sous-jacentes couvrent bien les cas d'accès.

Complexité : Faible · Temps : 30 min · Migration SQL requise

P1 IMPORTANT SEC-05 — XSS potentiel : document.write sans escapeHtml dans le popup archive RGPD assets/app.js:2028

Cause : La fonction d'impression de l'archive RGPD utilise document.write() pour générer un popup HTML. Les données ${nom}, ${archive.email}, ${archive.role}, ${archive.numero_mandat} sont interpolées directement sans appel à escapeHtml().

Impact : Si un nom ou email contenant des balises HTML (<script>alert(1)</script>) a été stocké en base, le code s'exécutera dans le contexte du popup lors de l'affichage de l'archive. Accès admin requis — risque conditionnel à la compromission d'un compte.

Impact métier : XSS stocké exploitable par un attaquant ayant accès à la DB ou via une injection SQL préalable. Risque de vol de session admin.

✔️ Fait vérifié statique — assets/app.js ligne 2028-2058
XSS stocké document.write admin only

Correction : Remplacer chaque interpolation directe par escapeHtml() :
${escapeHtml(nom)}, ${escapeHtml(archive.email)}, etc.

Complexité : Faible · Temps : 15 min · Risque régression : Nul

P2 MODÉRÉ SEC-06 — TOTP trust de 2h stocké en localStorage, contournable côté client assets/totp/totp.js

Cause : Après validation TOTP, un timestamp d'expiration est stocké dans localStorage (safe_totp_trust_<userId>). Ce mécanisme de "trust 2h" est purement client — un utilisateur peut modifier la valeur via DevTools pour prolonger le trust indéfiniment.

Impact : Pour les 3 opérations critiques (admin-create-user, delete-user, cancel-subscription), le contrôle AAL2 est effectué côté serveur dans la Edge Function → le contournement localStorage est inefficace. Mais pour les opérations dont la protection TOTP est uniquement côté client, un utilisateur peut bypasser la demande de code TOTP.

✔️ Fait vérifié statique — totp.js lignes 136, 161, 189, 287
⚠️ Hypothèse — liste complète des opérations protégées uniquement côté client non vérifiée
localStorage bypass TOTP client

Correction : Stocker le trust TOTP dans sessionStorage (effacé à la fermeture du navigateur) plutôt que localStorage. Pour les opérations sensibles, privilégier la vérification AAL2 côté serveur.

Complexité : Faible · Temps : 30 min

P2 MODÉRÉ SEC-07 — 23 politiques RLS avec USING(true) — accès illimité pour tout utilisateur authentifié Supabase DB

Cause : 23 tables (modules Click & Collect, Cyber, DPO, SEO, Social) ont des politiques RLS de type ALL ... USING (true) WITH CHECK (true). Cela signifie que tout utilisateur authentifié peut lire, écrire, modifier et supprimer toutes les données de ces tables, sans restriction de propriété.

Exemples : cc_client_profiles, cc_commandes, cyber_client_audits, dpo_client_profiles, social_posts…

Impact : Un DCI peut accéder aux données clients d'un autre DCI dans ces modules. Violation du principe de moindre privilège et de séparation des données clients.

✔️ Fait vérifié — advisors sécurité Supabase (rls_policy_always_true, 23 occurrences)
USING(true) accès non restreint modules clients

Correction : Restreindre les politiques par propriétaire :
USING (created_by = auth.uid()) ou USING (user_id = auth.uid()) selon la table.

Complexité : Moyenne · Temps : 3-5h · Migration SQL par table

P2 MODÉRÉ SEC-08 — GitHub Actions non épinglées par SHA (risque supply chain) .github/workflows/

Cause : Les actions tierces sont référencées par tag (@v5, @v6) plutôt que par SHA de commit immuable. Si le repo d'une action est compromis et que son tag est modifié, du code malveillant s'exécuterait dans le pipeline CI.

Actions concernées : actions/checkout@v5, actions/setup-node@v6, actions/upload-pages-artifact@v3, actions/deploy-pages@v4, denoland/setup-deno@v2.

✔️ Fait vérifié statique — .github/workflows/*.yml
supply chain mutable tags

Correction : Épingler par SHA :
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v5

Complexité : Faible · Temps : 30 min · Script scripts/check-sri.sh peut être adapté

P2 MODÉRÉ SEC-09 — Source maps générés en production (exposition du code source) build.mjs

Cause : build.mjs génère des source maps (sourcemap: true) qui sont inclus dans les artefacts déployés sur GitHub Pages. Les DevTools du navigateur permettent alors de lire le code source original non minifié.

Impact : Expose la logique métier, les noms de tables Supabase, les flux d'authentification et les endpoints aux yeux de tout utilisateur. Facilite la rétro-ingénierie.

✔️ Fait vérifié statique — build.mjs
source maps exposition code

Correction : Désactiver les source maps en production :
sourcemap: process.env.NODE_ENV !== 'production' ? 'linked' : false

Complexité : Faible · Temps : 10 min

P2 MODÉRÉ SEC-10 — Protection contre les mots de passe compromis désactivée (HaveIBeenPwned) Supabase Auth

Cause : L'option leaked_password_protection est désactivée dans la configuration Supabase Auth. Supabase peut vérifier les mots de passe lors de la création ou du changement contre la base HaveIBeenPwned (HIBP) pour rejeter les mots de passe connus comme compromis.

Impact : Un utilisateur peut définir un mot de passe présent dans des bases de données de fuites (ex : Password123!), augmentant le risque de compromission par credential stuffing.

✔️ Fait vérifié — advisors sécurité Supabase (auth_leaked_password_protection)
credential stuffing auth

Correction : Activer dans Supabase Dashboard → Authentication → Policies → Enable HaveIBeenPwned password protection.

Complexité : Faible · Temps : 5 min · Action manuelle Dashboard

P2 MODÉRÉ SEC-11 — 16 fonctions avec search_path mutable (risque injection de schéma) Supabase DB

Cause : Des fonctions comme trg_update_profil_complet, flag_profils_incomplets, update_cyber_audits_updated_at… n'ont pas de SET search_path fixe. Un attaquant avec accès à la DB pourrait créer un schéma malveillant qui intercepte les appels.

✔️ Fait vérifié — advisors sécurité Supabase (function_search_path_mutable, 16 occurrences)
search_path injection schéma

Correction : Ajouter SET search_path = public à chaque fonction dans une migration SQL.
Référence : supabase.com/docs/guides/database/functions#security

Complexité : Faible · Temps : 1h · Migration SQL

P3 DETTE SEC-13 — Table audit_logs sans aucun index de recherche Supabase DB

Cause : La table audit_logs n'a aucun index sur les colonnes de requête les plus fréquentes : user_id, action, created_at, entity_type, entity_id.

Impact : Scan séquentiel complet à chaque consultation du journal d'audit. Performance dégradée dès quelques milliers d'entrées. La traçabilité est un élément central de la conformité RGPD.

✔️ Fait vérifié dynamique — execute_sql sur pg_indexes

Correction migration SQL :
CREATE INDEX idx_audit_logs_user ON audit_logs(user_id);
CREATE INDEX idx_audit_logs_action ON audit_logs(action);
CREATE INDEX idx_audit_logs_created ON audit_logs(created_at DESC);

Complexité : Faible · Temps : 15 min

P3 DETTE SEC-14/15/16 — Dette performance DB : 44 FK sans index, 73 RLS init plan, 25 index inutilisés Supabase DB

SEC-14 — 44 clés étrangères sans index couvrant : Les JOINs sur les FK non indexées provoquent des scans séquentiels. Exemples : archived_users.archived_by_fkey, contracts.created_by_fkey, chartes_usage_si.user_id_fkey…

SEC-15 — 73 politiques RLS avec auth_rls_initplan : Les politiques utilisant auth.uid() / auth.role() sans (select auth.uid()) réévaluent la fonction à chaque ligne scannée. Impact significatif sur les tables volumineuses.

Fix SEC-15 : Remplacer auth.uid() par (select auth.uid()) dans les politiques RLS pour forcer l'évaluation une seule fois.

SEC-16 — 25 index non utilisés : idx_notif_created, idx_leads_email, idx_leads_etape… Overhead en écriture sans bénéfice en lecture.

✔️ Fait vérifié — advisors performance Supabase (159 entrées)

Complexité : Moyenne · Temps : 4-6h · Migration SQL par batch

P3 DETTE SEC-17 — npm audit absent du pipeline CI .github/workflows/deploy.yml

Cause : Aucune étape npm audit ou outil d'analyse de dépendances (Dependabot, Snyk) n'est présent dans le workflow CI. La seule dépendance (esbuild) est peu risquée, mais le principe s'applique à tout ajout futur.

✔️ Fait vérifié statique — deploy.yml

Correction : Ajouter après npm install :
- name: Audit sécurité npm
  run: npm audit --audit-level=high
